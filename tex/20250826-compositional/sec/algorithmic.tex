\section{Algorithmic typing}
\label{sec:algorithmic-typing}

This section develops an algorithmic version of the type system in
\cref{sec:types}. The algorithmic system is directly amenable to
construct a type checking algorithm.



\subsection{Typing relation}

Algorithmic typing requires an extended grammar of session types
$\AS$ that includes unification variables $\TVar$ drawn from
$\TVAR$. We write $\AT$ for types that refer to $\AS$ instead of $\S$.

Algorithmic typing collects a set of constraints $\CSet$. Constraints
come in three forms: 
\begin{itemize}
  \item equality constraints between two types $\AT \CEq \ATU$;
  \item mobility constraints $\CMobile \AT$;
  \item boundedness constraints $\CBounded \AS$.
\end{itemize}

\begin{definition}[Closing substitution]
  A substitution $\sigma : \TVAR \to \AS$ is a \emph{closing
    substitution} for  a set of constraints $\CSet$ if for every
  unification variable $\TVar$ in $\CSet$, $\sigma (\TVar) \in \S$ and all
  constraints are respected.
\end{definition}

\Cref{fig:typing-algorithmic} presents the algorithmic typing
judgments, which are based on bidirectional typing~\cite{DBLP:journals/csur/DunfieldK21}. The
synthesis judgment $\InferType \E \AT \Eff$ takes as input the context $\Ctx$
and expression $\E$. It computes the expression's type $\AT$, its effect
$\Eff$, and a set of constraints $\CSet$. The checking judgment $\CheckType \E
\AT \Eff$ takes as input the context $\Ctx$, expression $\E$, and type $\AT$. It
returns the expression's effect $\Eff$ as well as a set of constraints
$\CSet$.

\begin{definition}[Context join]
  The operator $\_ \CtxJoin \_$ joins two contexts according to the
  direction $\Dir$.
  %
  \[ \Ctx[1] \CtxJoin \Ctx[2] = \begin{cases}
        \Ctx[1] \CtxPar \Ctx[2] & \text{if $\Dir \in \{ \DirUnord, \DirNone \}$} \\
        \Ctx[1] \CtxSeq \Ctx[2] & \text{if $\Dir = \DirLeft$} \\
        \Ctx[2] \CtxSeq \Ctx[1] & \text{if $\Dir = \DirRight$}
     \end{cases}
  \]
  %
\end{definition}

\begin{definition}[Context restriction]
The operator $\_|_{X}$ restricts a context to a set of variables $X$%.
%
\[
  \CtxRestrict*{X} = \begin{cases}
    \CtxRestrict*[1]{X} \CtxPar \CtxRestrict*[2]{X} & \text{if $\Ctx=\Ctx[1] \CtxPar \Ctx[2]$} \\
    \CtxRestrict*[1]{X} \CtxSeq \CtxRestrict*[2]{X} & \text{if $\Ctx=\Ctx[1] \CtxSeq \Ctx[2]$} \\
    \CtxVar \EVar \T & \text{if $\Ctx=\CtxVar \EVar T$ and $\EVar \in X$} \\
    \CtxEmpty & \text{otherwise}
  \end{cases}
\]
%
\end{definition}

\begin{lemma}[Soundness]
  If\/ $\CheckType \E \AT \Eff$ and there is a closing substitution $\sigma$
  such that $\Fv(\AT) \subseteq \dom(\sigma)$ then $\HasType \Ctx \E {\AT{[\sigma]}}$.
\end{lemma}

\begin{lemma}[Completeness]
  If\/ $\HasType \Ctx \E \T$ then $\CheckType \E \AT \Eff$ for some type $\AT$, set of
  constraints $\CSet$, and closing substitution $\sigma$ such that $\T=\AT{[\sigma]}$.
\end{lemma}


\subsection{Constraint solving}

Potential solutions can be checked by the equivalence algorithm for context
free session types. However, the literature does not provide unification
algorithms. \pt{This must be made more precise! If I read the
  wikipedia article correctly, then solutions can be enumerated by
  one-sided paramodulation; so the problem should be semi-decidable,
  at least.}
Our approach is based on the fact that unification variables are
only introduced through split operations. As every session
type is linear, if a term with type $\TVar$ is not used the only viable solution for $\TVar$ is
$\SSkip$. Otherwise, we can infer the concrete type for $\TVar$ the terms use.
Multiple uses in different branches of the program must yield equivalent types.
If there are no solutions, the program is ill-typed.

any session type $\S$ such that $\Skips\S$ are solutions for unification
variables.
