
\section{Motivation}
\label{sec:motivation}

Context-free session types (CFSTs) extend regular session types with a
general sequential composition operator. A type $S_1;S_2$ describes a
session that first behaves as $S_1$ and then continues as $S_2$. 
Composition is associative with unit \lstinline{Skip}, the inactive session.
% , that is, \lstinline{Skip};$S = S =
% S$;\lstinline{Skip} and $(S_1;S_2);S_3 = S_1;(S_2;S_3)$.
Thus
\lstinline{Skip} and \lstinline{;} form a monoid up to type
equality. The types \lstinline{Close} and \lstinline{Wait} indicate
(active and passive) termination of a session.

Sequential composition also interacts with choice in the expected way.
Appending a continuation to a choice appends it to every branch:
$(\oplus\{\ell_i:S_i\}_{i\in I});T =
\oplus\{\ell_i:S_i;T\}_{i\in I}$. The same law holds for external
choice. These equations make non-tail-recursive protocols
usable as session types: a continuation can be pushed through
the structure of a type until it reaches the leaves of a protocol.

As usual for session types, channels are linear resources. A channel
has to be used exactly once and every communication operation consumes
the channel at its current type and produces the channel at its
continuation type. Linearity is what makes the session type track the
state of the protocol, but it also forces programs to thread channels
explicitly from one operation to the next.

Additional remarks on design choices in this section, including why
some operations remain in resource-passing style, are given in the
supplement in \cref{sec:remarks}.

\subsection{Borrowing from Context-Free Session Types}
\label{sec:borr-from-cont}


Context-free session types enable writing elegant code for
serializing tree structures when communication is restricted to
primitive values.
% \footnote{We assume session-typed communication over a network
%   or between sandboxed processes, where only primitive data can be
%   passed, but no pointers.}
Consider an example from \citet{DBLP:conf/icfp/ThiemannV16} for serializing binary trees of type \lstinline{Tree}.
\begin{lstlisting}
type Tree = Leaf | Node int Tree Tree
\end{lstlisting}
Here is the corresponding session type \lstinline{TreeChan}:
\begin{lstlisting}
type TreeChan = oplus{ Leaf: Skip, Node: !int; TreeChan; TreeChan }
\end{lstlisting}

The type \lstinline{TreeChan} is called context-free because it is
recursively defined and the recursion is not restricted to tail
positions. The \lstinline{Node} alternative is an example where the
\lstinline{TreeChan} type refers two times in a row to itself. The
first occurrence of \lstinline{TreeChan} in this
type is illegal in regular session type systems (e.g.,
\cite{DBLP:journals/jfp/GayV10,DBLP:conf/esop/LindleyM15,Gay_Vasconcelos_2025}) because
they syntactically restrict recursion to tail position. Context-free
session types lift this restriction as their composition operator $S_1;S_2$ connects
two sessions sequentially: the composed session comprises the
actions of $S_1$ followed by the actions of $S_2$.
% \footnote{One
%   might argue that CFSTs are close to Honda's original vision for
%   session types \cite{DBLP:conf/concur/Honda93}, which also features
%   symmetric composition.}

\Cref{lst:cfst-api} displays the API of CFSTs. The \lstinline{send},
\lstinline{recv}, and \lstinline{select} operations are standard for
sending a message, receiving one, and selecting an alternative in a
branching session type. The \lstinline{branch} operation maps a channel
offering different labeled alternatives to a variant type with the
same labeled alternatives. Standard pattern matching suffices to
decode the chosen alternative and continue.
Given the types and the CFST API, we can implement the serialization function:
\begin{lstlisting}[numbers=left]
sendTree : Tree -> forallsigma.TreeChan;sigma -> sigma
sendTree Leaf c =
  select Leaf c
sendTree (Node x l r) c =
  let c1 = select Node c   (* c1 : !int; TreeChan; TreeChan; sigma *)
      c2 = send x c1       (* c2 : TreeChan; TreeChan; sigma *)
      c3 = sendTree l c2   (* c3 : TreeChan; sigma *)
      c4 = sendTree r c3   (* c4 : sigma *)
  in  c4
\end{lstlisting}
However, there are several catches. First, the channel \lstinline{c}
is passed explicitly throughout the code; a pattern we call
\emph{resource-passing style}. This style forces the programmer to
make the function polymorphic on the continuation type of the channel,
which is \lstinline{sigma} in this example.
Second, the \lstinline{Node} branch of \lstinline{sendTree} relies on
polymorphic recursion. To see this, consider the type of
\lstinline{c2} in line 6--7. To invoke \lstinline{sendTree}
recursively, we need to instantiate the type argument
\lstinline{sigma} with \lstinline{TreeChan; sigma} rather than just
\lstinline{sigma}, which is the
telltale for polymorphic recursion.

Our new language {\thecalculus} amends these problems. The type
definitions remain unchanged, but the implementation of
\lstinline{sendTreeB} relies on the session API with borrowing shown in
\cref{lst:bcfst-api}: the operations \lstinline{send} and \lstinline{recv} adopt
the borrowing style, but \lstinline{select} and \lstinline{branch}
remain in resource-passing style (see \cref{sec:remarks} for the
reason). To make this switch apparent, we present code
using the \thecalculus API with grey background and golden keywords.
\lstset{style=eclipse-new}
\begin{lstlisting}[numbers=left]
sendTreeB : Tree -> TreeChan -> Skip
sendTreeB Leaf c =          (* c : TreeChan *)
  select Leaf c
sendTreeB (Node x l r) c =
  let c = select Node c in  (* c : !Int; TreeChan; TreeChan *)
  send x &c;                (* c : TreeChan; TreeChan *)
  sendTreeB l &c;           (* c : TreeChan *)
  sendTreeB r &c
\end{lstlisting}
What has changed? First, the type of \lstinline{sendTreeB} is no longer polymorphic over the
continuation type. Instead, the channel argument \lstinline{c} starts
out at type \lstinline{TreeChan} without specifying a
continuation. The return type is \lstinline{Skip}, which indicates an inactive
channel. 
Second, there is no explicit resource
passing, except for \lstinline{select}. Each use of the borrow notation
\lstinline{&c} implicitly splits off a prefix of the session type
according to the function argument and updates \lstinline{c} to the
suffix. To see this, consider the type of \lstinline{c}---after
splitting off the borrow in the corresponding line---shown in the
comments. Before the \lstinline{send} operation (line 5--6), the type
of \lstinline{c} starts with \lstinline{!Int; ...}. ``Taking the
borrow'' splits off the borrowed type \lstinline{!Int}---the
argument type of \lstinline{send x}---and
updates \lstinline{c} to the type shown in line 6. As the type of
\lstinline{c} is itself a borrowed type, one could speak of a
\emph{reborrow}. 

Third, as there is no polymorphism, there is no polymorphic
recursion, either.

Finally, the channel variable \lstinline{c} can be used in a
non-linear way, as long as its uses are under borrows, as in
\lstinline{&c}. The understanding is that the borrows are taken and
processed according to the evaluation order of the program. This view does
not abandon linearity. Rather, the type system internalizes the
linear use of the channel: each borrow consumes a distinct prefix of
the current session type and leaves the residual type for the next use
of \lstinline{c}.


\subsection{Local and Remote Borrowing}
\label{sec:local-remote-borr}

\Cref{sec:borr-from-cont} demonstrated
 \emph{local borrowing}, where the borrows remain in the same
process and they are processed sequentially, following the evaluation order.
In this local setting, operations on borrows are
cheap: borrowing merely names the next prefix of the same channel and
leaves the suffix for the surrounding computation. No other process can
race with the continuation, so the evaluation order of the program and
the left-to-right order enforced by typing already provide the required
sequencing.

However, in some situations we want to send a borrow to
another process. As an example, suppose a server-side renderer streams
HTML nodes to a client instead of constructing a DOM tree in
memory first. Here is a protocol for such a renderer.
\begin{lstlisting}
type HtmlChan  = oplus{ Text: !String, Elem: !String; ChildChan }
type ChildChan = oplus{ Done: Skip,    Child: HtmlChan; ChildChan }
\end{lstlisting}
The \lstinline{Elem} alternative first sends a tag name, and then a sequence of children. The 
\lstinline{Child} alternative is context-free in the same sense as the
\lstinline{Node} alternative of \lstinline{TreeChan}: rendering one
child consumes an \lstinline{HtmlChan} prefix and then continues with
the rest of the child stream.

Now consider a renderer (\Cref{lst:rendering-example}) that emits a user panel whose first child is
computed by another process while the main process loads user
messages, say, from a database. We assume that
\lstinline{renderProf} and \lstinline{renderMsgArea} render
one HTML node on a channel of type \lstinline{HtmlChan; Drop} and
\lstinline{HtmlChan}, respectively. The type \lstinline{Drop}
is a new session type that comes with a companion type
\lstinline{Acq}. They indicate the placement of synchronization
operations that ensure correct handover of ownership from the borrowed session
to the session's suffix.
\begin{lstlisting}[numbers=left,float,caption={Rendering example},captionpos=b,label={lst:rendering-example}]
renderUser : User -> HtmlChan; Close -> Unit
renderUser user c =
  let c = select Elem c in
  send "div" &c;
  let c = select Child c in
  fork (lambda() -> renderProf user &c);
  (* c : Acq; ChildChan; Close *)
  let msgs = getMessages user in
  renderRest msgs (acquire c)

renderRest : Messages -> ChildChan; Close -> Unit
renderRest msgs c =
  let c = select Child c in
  renderMsgArea msgs &c;
  close (select Done c)

renderProf : User -> HtmlChan;Drop -> Unit
renderProf user c = ... drop c
\end{lstlisting}
The function \lstinline{renderUser} starts an HTML element and then
opens the child stream. It passes a borrow of the
first child position to a new process, which renders the user's profile.
The continuation renders the remaining children, here a messaging
area, and then closes the child stream. 

In this case, the borrow under the \lstinline{fork} is a \emph{remote
  borrow}. Such a borrow must be handled differently as operations on
the borrow are no longer necessarily ordered: the 
continuation must \emph{not} start emitting the second child before the
forked renderer has finished the first child. This synchronization is
expressed with the \lstinline{acquire} operation, which works on a
channel of type \lstinline{Acq; ...}. It blocks the
continuation until the borrowed prefix has been completed by the
forked process. Conversely, the forked process calls \lstinline{drop}
on a channel of type \lstinline{Drop} when it has finished working on
the prefix. Thus \lstinline{drop} and
\lstinline{acquire} form the synchronization point between the two
processes; the run-time mechanism that implements this point is
described in~\cref{sec:impl-local-remote}.

This setup is also reflected in the types that result from a remote
borrow. Line~7 shows the type of \lstinline{c}'s suffix, which starts
with an \lstinline{Acq}, and line~16 shows the type of the borrow,
which ends with \lstinline{Drop}. The types \lstinline{Acq} and
\lstinline{Drop} only arise from a remote split operation. The
\emph{only} operation on a channel of type
\lstinline{Acq; ...} is \lstinline{acquire} and similarly,
\lstinline{drop} only applies to a channel of type \lstinline{Drop}.

Next, we discuss a direct operational semantics of local and remote borrowing
before we move on to a low level implementation of the two concepts in
\cref{sec:impl-local-remote}.

\subsection{A Direct Semantics for Channel Borrowing}
\label{sec:direct-semant-chann}

\Cref{sec:borr-from-cont} already suggests an implementation strategy
for local borrowing. However, we start with an abstract
specification because it is easier to handle in a correctness
proof. The specification works on an elaborated core 
language, \thecalculus, where elaboration replaces borrowing with
explicit functional splitting 
operators. This translation is analogous to a translation in previous work
\cite{DBLP:journals/pacmpl/SaffrichS0V25}, which gives a formal
presentation along with a correctness proof. The novelty in our work
is the presence of \emph{two} operators, \lstinline{lsplit} for local
borrows and \lstinline{rsplit} for remote borrows. Borrows used in the
same process elaborate to \lstinline{lsplit} while borrows used across
\lstinline{fork} or sent to another process elaborate to
\lstinline{rsplit}.  Our discussion starts with the local variant and 
we show the translation of \lstinline{renderRest} in
\thecalculus in \cref{fig:local-split-program}.

Execution proceeds by repeatedly exposing the next local prefix of the
channel sequence. \Cref{fig:local-split-trace} shows a suffix of its reduction
sequence.
The \lstinline{nu[c][d] in ...} operator is a channel restriction as known
from session calculi \cite{Gay_Vasconcelos_2025}. It introduces a
communication channel with endpoint names \lstinline{c} and 
\lstinline{d}. We generalize channel restriction so that each
compartment \lstinline{[c]} and \lstinline{[d]} contains a
\emph{binder group}: a list of channel names and separator markers.
Its working is best illustrated with the example trace.

To keep the trace short (\cref{fig:local-split-trace}), the
reduction starts at the local suffix in \lstinline{renderRest}.

The split on \lstinline{c2} following the \lstinline{select} \emph{replaces} the
binder for \lstinline{c2} with two fresh channel names \lstinline{c3}
and \lstinline{c4}. That is, both compartments of the \lstinline{nu}
contain a flat list of channel names, and all operations work on these
channel names from left to right. To ensure the correct
sequencing, the semantics does not allow any action on \lstinline{c4} before
\lstinline{c3} has been consumed. This restriction is supported by
typing: the \lstinline{lsplit} operation returns an \emph{ordered
  pair}, whose components must be processed and consumed in
left-to-right order.
% The \lstinline{drop} operation (not used in this
% example) applied to some \lstinline{c} just discards \lstinline{c} if
% it is the first channel name in the compartment.

On the client side, in the process $P_d$ that holds the other
channel end \lstinline{d}, 
splitting can be performed on \lstinline{d} independently of
the splitting on \lstinline{c}. The only constraint is that, in the
end, when \lstinline{close} and \lstinline{wait} calls match, the
reduction requires that only one channel name is left in each
compartment. Under this condition, the channel can be closed and
eliminated. The figure does not show the transitions of the client $P_d$, which
are independent.

For remote borrowing, we incorporate the 
\lstinline{acquire} primitive and indicate a remote split with the
symbol ``\lstinline{||}'' in the compartment. As an example, we consider the translation of
lines 5--7:
\begin{lstlisting}
  fork (lambda() -> renderProf user &c);
  let msgs = getMessages user in
  renderRest msgs (acquire c)
\end{lstlisting}
It translates to the following \thecalculus code.
\begin{lstlisting}
(* initially                  c : HtmlChan; ChildChan; Close *)
let c1, c2 = rsplit c in   (* c1 : HtmlChan; Drop *)
                           (* c2 : Acq; ChildChan; Close *)
fork (lambda() -> renderProf user c1);
let msgs = getMessages user in
renderRest msgs (acquire c2)
\end{lstlisting}
The translation inserts an \lstinline{rsplit} because \lstinline{c1}
and \lstinline{c2} are used in different processes. The
\lstinline{rsplit} returns an ordinary (linear) pair of channels
because synchronization is made explicit using the \lstinline{Acq} and
\lstinline{Drop} types.
Executing this code takes the steps shown  in
\cref{fig:remote-split-trace}  (ignoring the activity 
at the other end of the channel). The bracketed step abbreviates the
forked process running \lstinline{renderProf} until its final
\lstinline{drop c1}.
In summary, the \lstinline{rsplit} replaces one channel name by
\lstinline{[c1||c2]}, where ``\lstinline{||}'' marks a remote split. The
\lstinline{drop} is only enabled 
if its argument name comes last before a ``\lstinline{||}''.
The \lstinline{acquire} is only enabled if the preceding list segment has
been processed as in \lstinline{[||c2]}. Executing the \lstinline{acquire}
in the last step removes the exhausted segment and separator, yielding
\lstinline{[c2]}.

\subsection{Implementing Local and Remote Borrowing}
\label{sec:impl-local-remote}

We implement the high-level calculus by translation into
a low-level run-time session calculus with borrowing, \thertcalculus. The run-time calculus
is a concurrent 
lambda calculus with the usual session-style communication
primitives. However, \thertcalculus is untyped and it comes equipped with
asynchronous signaling primitives for implementing \lstinline{drop}/\lstinline{acquire}.
The direct semantics from~\cref{sec:direct-semant-chann} records ordering constraints in binding
groups. The run-time translation realizes the same constraints with
ordinary channel names and optional synchronization flags that
implement passing of ownership.

\Cref{fig:borrow-lifecycle-msc} summarizes the difference between local
and remote borrowing in the implementation.

Specifically, a channel is represented by a triple
$in \otimes c \otimes out$, where $c$ is a primitive channel name and
$in$ and $out$ are optional names of synchronization flags. The
incoming flag $in$ records whether the channel must be acquired before
it can be used. The outgoing flag $out$ records whether the channel
must be dropped after use. A synchronization binder
$(\varphi z \mapsto \phi)P$ binds a flag name $z$ in $P$ and stores
one of two states in $\phi$: \lstinline{drop} or \lstinline{acq}.
This flag $z$ implements the separator in
\lstinline{nu[c1||c2][...]}. It is the outgoing flag of the
borrowed prefix and the incoming flag of the suffix.

An \lstinline{rsplit} creates a flag in state
\lstinline{drop} and a freshly restricted channel sets its flag to \lstinline{acq}.
Intuitively, \lstinline{drop} means that the left, borrowed
part of a remote split is still active, \lstinline{acq} means that it
has been dropped and the right, suffix part may acquire.
This mechanism implements an ownership transfer of the
primitive channel from the borrowed part to the suffix.

Allocation creates a primitive channel pair and one synchronization
flag for each endpoint. The fresh endpoints carry incoming flags that
are already in the \lstinline{acq} state. Thus, an initial
\lstinline{acquire} is required to reduce each channel to normal
operation, removing the flag from the 
channel triple by replacing it with \lstinline{*}. Once these optional
flag components are absent, communication is just ordinary
session communication: \lstinline{send}/\lstinline{recv},
\lstinline{select}/\lstinline{branch}, and \lstinline{close}/\lstinline{wait}
all operate directly on the primitive channel name $c$.

\Cref{fig:borrow-lifecycle-msc-local} shows the lifecycle of a local borrow. The rule for
\lstinline{lsplit} returns two copies of the same run-time channel
triple. No new channel is allocated and no synchronization flag
is introduced. This simple approach has no overhead and is sound
because the source type system has already established the left-to-right use of the two pieces. The
run-time representation can therefore be imperative: the two pieces are
aliases for the same underlying channel object, and using the first
piece advances that object to the state where the second piece starts.

Remote splitting is the only place where new synchronization
machinery is introduced
(\cref{fig:borrow-lifecycle-msc-remote}). Splitting a channel
$in \otimes c \otimes out$ remotely creates a fresh flag $z$ in state
\lstinline{drop} and returns two channel triples:
$in \otimes c \otimes z$ for the borrowed prefix, and
$z \otimes c \otimes out$ for the suffix. The prefix therefore carries
an outgoing obligation to call \lstinline{drop}; the suffix carries a
matching incoming obligation to call \lstinline{acquire}. Calling
\lstinline{drop} changes the flag from \lstinline{drop} to
\lstinline{acq} and returns immediately, so the notification is
asynchronous. A later \lstinline{acquire} on the suffix can fire only
after this state change and removes the synchronization
flag. Thus a remote split pays for exactly the
synchronization needed to coordinate the two processes operating on
the channel, whereas a local
split pays nothing beyond sharing the channel representation.
