\section{Algorithmic Typing}
\label{sec:algorithmic-typing}

This section develops an algorithmic version of the type system in
\cref{sec:types}. The algorithmic system is directly amenable to construct a
type checking algorithm. It employs a sound heuristic for inferring the
remainder session types of split operations. First, a set of constraints is
collected involving fresh unification variables. In a second step, the unification variables
are solved and validated with respect to the constraints. The constraint
generation is part of the algorithmic typing judgements
in \cref{sec:alg-typing-relation}. Constraint solving is described in
\cref{sec:constraint-solving}.


\subsection{Algorithmic Judgments}
\label{sec:alg-typing-relation}

Algorithmic typing requires an extended grammar of session types
$\AS$ that includes unification variables $\TVar$ drawn from
$\TVAR$. We write $\AT$ for types that refer to $\AS$ instead of $\S$.

Algorithmic typing collects a set of constraints $\CSet$. Constraints $C$
come as either equality constraints between two types, or mobility constraints:
$
  C \grmeq* \AT \CEq \ATU \grmor\CMobile \AT
$.

\begin{definition}
  A substitution $\sigma : \TVAR \to \AS$ is a \emph{closing
    substitution} for  a set of constraints $\CSet$, written $\sigma
  \models \CSet$, if for every variable $\TVar$ in $\CSet$, $\sigma (\TVar) \in \S$ and all
  constraints are respected.
\end{definition}

\Cref{fig:typing-algorithmic} presents the bidirectional algorithmic
typing judgments~\cite{DBLP:journals/csur/DunfieldK21}. The synthesis
judgment $\InferType \E \AT \Eff$ takes as input the context $\Ctx$
and expression $\E$. It computes the expression's type $\AT$, its
effect $\Eff$, and a set of constraints $\CSet$. The checking judgment
$\CheckType \E \AT \Eff$ takes as input the context $\Ctx$, expression
$\E$, and type $\AT$. It returns the expression's effect $\Eff$ as
well as a set of constraints $\CSet$. Judgements make use of context
join $\Ctx[1] \CtxJoin* \Ctx[2]$, which joins two contexts according to
the direction $\Dir$, and context restriction $\CtxRestrict*{X}$,
which restricts a context to a set of variables $X$; the definitions
are given in \cref{fig:context-operators}. Type
normalisation~\cite{DBLP:journals/iandc/AlmeidaMTV22}, written
$\NormaliseType {\T[in]} {\T[out]}$ (see
\cref{fig:typing-normalisation}), exposes the top-level type
constructor in synthesis judgements.

We have checking rules for function definitions, pairs, sum
injections, and checking against a synthesized type. They check
against normalized types. For lambdas,
\RuleName{A-Abs} enforces the supplied function type, which specifies
the mobility $\Mob$, the direction $\Dir$, and the effect $\Eff$. If
$\Mob = \MobMobile$, then all free variables in the lambda's body must
be mobile. The direction enforces the way the new binding is joined
to the context. The effect puts an upper bound on the effect of the
body. The mobility constraints are inserted into the outgoing
constraint set $\CSet$. \RuleName{A-AbsRec} is similar, but specialized to a recursive
function, which is unrestricted and hence mobile.
For pairs, \RuleName{A-Pair} enforces the ordering of the pair; for
injections, \RuleName{A-Inj} specifies the types of the remaining
summands. Rule \RuleName{A-Check} switches between checking and
synthesis as usual, but leaves checking the equality of the inferred
type $\TU$ against the checked type $\T$ as a constraint.
The other rules (and most synthesis rules) just gather constraints
from the subterms and propagate them.

In the synthesis rules, the context in the variable rule
\RuleName{A-Var} must subsume to a single variable binding and it must be
empty for most constants, then it normalises the inferred type. A type annotation
$\E:\T$ (\RuleName{A-Annot}) synthesizes $\T$ and checks $\E$ against $\T$, then normalises the type. Rule
\RuleName{A-LSplit} asks for $\CSplit$ to be annotated with the
borrowed type ${\S}$. This annotation defines the output type
$\TArr{\MobMobile}{\DirUnord,\EffPure} {\SSeq\S\TVar}
{\TPairOrd{\S}{\TVar}}$ up to the continuation, which is supplied by a
fresh variable $\TVar$. This rule does \emph{not} add a
constraint. The constraint appears due to the interplay of rules
\RuleName{A-App} and \RuleName{A-Check}: When $\CSplit_{\S}$ is
applied to an argument, the type of the argument is checked against
$\SSeq\S\TVar$. Solving the equation generated by this check resolves
the assignment to $\TVar$.

Rule \RuleName{A-RSplit} is similar. The rules \RuleName{A-Select}
and \RuleName{A-Branch} normalise the type of the argument before checking it is the appropriate choice type.
Function application
\RuleName{A-App} has to cater to two
different styles of function application, which differ in the
evaluation order of their subterms given by some $\Dir \in \{\DirLeft,
\DirRight\}$. The inferred function type's direction $\Dir'$ is
compatible with $\Dir$ if $\reqDir (\Dir', \Dir)$ holds: All
functions, except $\DirRight$ functions are compatible with
left-to-right evaluation. The function's direction $\Dir'$ also
determines which subterm can have effects as prescribed by $\reqEff
(\Dir', \Eff[1], \Eff[2])$.
\begin{align*}
  \reqDir & = 
  \{(\DirLeft,\DirLeft),
    (\DirRight,\DirRight),
    (\DirUnord,\DirRight),
    (\DirNone,\DirRight)\}
  &
  \reqEff & =
  \{(\DirLeft,\EffPure,\_),
    (\DirRight,\_,\EffPure)\}
\end{align*}
Apart from that, rule \RuleName{A-App} synthesizes the function type,
checks against the argument type and normalizes the return type. Sequential execution
requires that the sequence of the required contexts for the
expressions can be subsumed from the incoming context. Rule
\RuleName{A-LetPair} considers its subexpressions in sequence and
includes the variable binding in the context of the body according to
the order $\Dir$ of the pair.
Elimination of a sum type using \RuleName{A-Case} works similarly, but
in addition we have to add equations to $\CSet$ that force all branch
output types to be equal.

\subsection{Constraint Solving}
\label{sec:constraint-solving}
An algorithmic typing for an expression $\E$ yields a set of
constraints $\CSet$. In inference mode, the returned type $\AT$ may
refer to unsolved unification variables. Solving $\CSet$ boils down to a unification problem, but we are not
aware of a unification algorithm for the theory of context-free session
types, which involves associativity, neutral elements, and
distributivity. Hence, we
approach constraint solving by constraint simplification.

The equivalence constraints in $\CSet$ are simplified by structural
decomposition: pairs, sums, function types, and contexts are reduced
componentwise, while unit constraints disappear. The full rule set is given
in \cref{fig:constraint-simplification}. The rules are applied until a fixed point is reached. Constraints of the form $\TVar\CEq\AT$ provide a potential
solution for unification variable $\TVar$. If we are let with unification
variables occurring in $\CSet$ or $\AT$ for which no solution candidate
is found, the heuristic will fail and $\E$ is
considered ill-typed.

Potential solutions have to first be identified and then validated in respect to the whole set of
constraints $\CSet$.

We non-deterministically pick one candidate instantiation per variable, resulting in
a variable-free $\CSet*$\!. 
If there are multiple candidates for a unification variable the type to be substituted
into $\CSet$ can be picked non-deterministically. To see this, consider the case where after
simplification there are two constraints $\TVar \CEq \T$ and $\TVar \CEq \TU$. If
$\T$ is not a valid solution for $\TVar$, then $\TU$ is also not a solution.
The equality $\T \simeq \TU$ must hold for all types that can be selected. Since mobility
respects type-equivalence it is not possible to construct mobility constraints
that are satisfied by one but not the other candidate.

The mobility constraints are easily
validated because $\Mobile\T$ is decidable. 
The equality constraints can be verified because the equivalence of CFSTs is
decidable~\cite{DBLP:conf/tacas/AlmeidaMV20}. 
Because our type language is richer than ordinary CFSTs (e.\,g. the distinction
between left and right-ordered function types), we require
an equivalence respecting encoding from our types to CFSTs.
\Cref{sec:algorithmic-typing-appendix} describes the
translation and sketches a correctness proof for the encoding.

\subsection{Implementation}

We supplement our theoretical work on the bidirectional type checking algorithm from this section with an implementation written in Rust. 
It is based on the artifact \cite{saffrich25:_artif_borrow_from_session_types}
and uses FreeST 5.0
\cite{DBLP:journals/corr/abs-1904-01284,DBLP:journals/iandc/AlmeidaMTV22,
  almeida26:_frees_concur_progr_languag_with} for equivalence checking
of session types. The FreeST integration is described more closely in \cref{sec:appendix-implementation}.
The test cases of the Rust implementation include the motivational examples from \cref{sec:motivation}.
